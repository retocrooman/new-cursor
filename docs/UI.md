# UI

- 画面構成
  - 3 カラムシェル：左タスクリスト | 中央タスク詳細 | 右司令官チャット
  - エージェントごとのカラム分割はしない（担当の可視化はタイムライン上の run で行う）
- タスクリスト
  - 親タスクのみ表示する（子タスクは親のタイムラインに載せる）
  - 行は 2 行のスリム表示とする
    - 1 行目：タイトル + 工程アイコン（stage）
    - 2 行目：レポジトリ名 · 最後のイベント
  - 実行時間・消費トークン数はリストには出さない（run タイムラインで見る）
- タスク詳細
  - 表示順（上から）
    1. 工程パイプライン（stage pipeline）— MVP の 6 工程のみ。現在位置を示す
    2. 背景・検証 — `background` と `verificationItems` を 1 セクションにまとめて表示
    3. 意思決定 — `task_decisions` テーブルの一覧（後述）
    4. 統合タイムライン — ドメインイベント・run・子タスク起票を時系列で混在表示
    5. 予測イベントフロー（predicted event flow）— 縦方向。未到来の工程は破線
  - 表示しないもの
    - 終了条件セクション — 工程パイプラインと重複するため MVP では省略
    - 担当エージェントセクション — 購読設定は担当割当ではない。エージェント名は run 行に出す
- 統合タイムライン
  - task / run / event の区別
    - **イベント** — 司令室が記録した状態変化（`task.created` など）
    - **run** — Worker の 1 回の SDK 実行。エージェント名・トークン消費を表示
    - **子タスク** — エスカレーション等で起票された子を、別セクションではなくタイムライン 1 行として表示
  - 詳細は [EVENTS.md](./EVENTS.md) のドメインイベント一覧を参照
- 意思決定リスト
  - イベント履歴からは組み立てない
  - `task_decisions` テーブル由来：エージェントが記録した判断メモ + ユーザーへの質問と回答
  - Commander および Worker が `decisions.create` / `record_decision` JSON で追記する
- 予測イベントフロー（タスク詳細内）
  - 現在 stage 以降の典型連鎖を縦に表示する
  - 未到達の工程は破線で示す
  - PR マージなど将来拡張予定の工程は、パイプラインには含めずここだけ破線で示す
  - 汎用オートメーション用の「イベントフロー編集画面」とは別物（下記）
- 司令官チャット
  - Cursor 風のチャット UI（メッセージ履歴 + 入力欄）
  - 送信ボタン、モデルピッカー（AgentModelPicker）を備える
  - リストで選択中のタスク文脈をプロンプトに渡す
  - 起票時は、タスクタイトルを必ずユーザーに確認する
  - 起票時は、ベースブランチを必ずユーザーに確認する
  - 起票時は、ブランチ名を必ずユーザーに確認する
  - 起票時は、背景を必ずユーザーに確認する
  - 起票時は、検証項目を必ずユーザーに確認する
  - 確認した背景・検証項目は `task.created` とともに task 行へ永続化する
- イベントフロー（汎用オートメーション）
  - 目的：任意のイベント連鎖（いつ・何が起きたら → 何をする → 次に何を送る）を定義する汎用オートメーション画面。タスク工程テンプレートに限定しない。
  - 遷移：3 カラムの運用 UI とは別画面。ナビゲーションから開く。
  - トリガー：イベント種別と条件（発火条件）を指定する。
  - アクション：実行内容（通知・API 呼び出し・状態更新など）を指定する。
  - 発火：アクション完了後に送る次イベントを指定する。
  - 保存：定義は司令室（API+DB）に保存し、イベント駆動のルーティングに使う。
  - 連携：トリガー条件を満たしたとき、司令室が SQS 経由で Worker に作業依頼を届ける。
- UI 変更時の確認
  - `apps/web` の UI 変更は [ui-browser-verification](../.cursor/rules/ui-browser-verification.mdc) に従い、cursor-ide-browser MCP で動作確認を受け入れ項目に含める
