# 合意点

- タスクの進め方
  - 固定工程：MVP は 6 stage（`created` → `worktree_requested` → `worktree_ready` → `queued` → `implementing` → `completed`）。PR マージなどの工程拡張は予定だが、MVP パイプラインには含めない（UI の予測イベントフローでは破線表示）
  - イベントフロー：工程を横断する汎用オートメーション。固定工程と併用する
  - 作業場所：タスクごとに git worktree を用意して作業する
  - ブランチ確認：開始時と完了前にブランチ状態を確認する
  - コンフリクト解消：ブランチの競合を検知し、解消してから次に進める
  - PR 作成：実装後に変更をコミットし、PR を作成する
  - CI 連携：CI の結果をイベントとして受け取り、次の処理につなげる
  - 並列実行：複数タスクを同時に進められる。同一リポジトリ・同一ブランチの競合は直列キューで処理する
  - 記録：試行回数と経緯を残し、あとから追えるようにする
  - エスカレーション：親タスクはブロックし、問題用の子タスクを起票する（`task.escalation.requested` 等のドメインイベント）。子タスク完了後に親を再開する
  - 親子タスク：それぞれ独立した工程パイプラインを持つ。子は親詳細の別セクションではなく、統合タイムラインの 1 エントリとして表示する
  - 承認：MVP は UI ボタンで承認し、PR は auto-merge する
  - 起票時の永続化：`background` と `verificationItems` は `task.created` と同時に task 行へ保存する
- 実行単位
  - Worker の作業単位は 1 run = Local Cursor SDK 1 回の呼び出しとする
  - run ごとにエージェント・トークン消費を記録し、UI タイムラインで表示する
- 意思決定（decisions）
  - イベント（ES/CQRS）とは別ドメイン。`task_decisions` テーブルに保存する
  - 用途：エージェントの判断に迷った箇所のメモ、ユーザーへの質問とその回答
  - 記録経路：Commander と Worker が `decisions.create` / `record_decision` JSON アクションで追記する
  - UI の意思決定リストはイベント履歴から組み立てない
- イベント駆動
  - 配信基盤：Postgres と SQS のハイブリッド構成とする
  - アウトボックス：司令室がトランザクション内で outbox に書き、relay が SQS へ配信する
  - インボックス：Worker が SQS から受け取り、冪等処理する
  - relay：outbox を ElasticMQ に publish する独立プロセスとする
  - ローカル SQS：ElasticMQ を使う
  - ルーティング：イベント購読（イベント種別とペイロード）で、どのエージェントが処理するか決める
  - 購読条件：エージェントごとに購読するイベント種別やフィルタを設定する
  - 複数担当：同じイベントを複数エージェントが購読できる。SQS で 1 件 1 消費する
  - 購読と担当：購読設定は「このイベントを処理する候補」であり、タスクへの固定担当割当ではない
  - 状態遷移：承認・PR マージ・CI 完了などのイベントで次の処理を決める
  - 待機と再開：待機中は状態を保存し、きっかけが来たら同じタスクを再開する
  - 定期実行：cron でセキュリティ診断・パフォーマンス診断・リファクタ提案を起動する
- エージェントとルール
  - ラベル付与：1 エージェントに複数ラベルを付けられる
  - エージェント作成：ユーザーが自由に追加できる
  - ルール保存：ルールはラベル単位で DB に保存する
  - ルール取得：プロンプト実行前に DB をクエリし、必要なルールを取得する
  - ルール注入（共通）：`all` ラベルのルールを、全エージェントのプロンプトに毎回入れる
  - ルール注入（個別）：エージェントのラベルに一致するルールを、そのプロンプトに毎回入れる
- 自己改善
  - 実行中の注入：タスク実行中も必要に応じて DB からルールを取得し、プロンプトに注入する
  - 終了後の振り返り：タスク終了後に振り返りを行い、次回以降の実行に反映する
  - 定期改善：cron で実行記録を見直し、ルールと注入方法を改善する
- 連携・拡張
  - 業務アプリ：GitHub・Slack・Confluence など、あらゆる業務アプリをつなぐ
  - プラグイン：処理・検証・通知・作業場所・MCP を用途ごとに差し替える
  - 複数リポジトリ：複数リポジトリのタスクを一元管理する（1 タスク＝1 リポジトリ）
  - 外部リポジトリ：登録時に clone し、worktree はホストの WORKTREE_ROOT 上に作る
  - MCP：工程ごとにエージェントが使う外部ツールを切り替える
- アーキテクチャ
  - モノレポ：monorepo-template を fork せず本リポジトリへ移行する
  - ビルド：Turborepo + pnpm、パッケージ scope は `@new-cursor/*` とする
  - apps/web：Next.js + oRPC で司令室 API と UI を提供する
  - apps/worker：TypeScript、ホスト上で `pnpm dev` 起動する（コンテナ化しない）
  - apps/relay：outbox を SQS へ配信する独立プロセスとする
  - ローカル起動：docker-compose は DB と ElasticMQ のみ。web・worker・relay はホストで動かす
  - 認証：MVP は better-auth を使う
  - ドメイン：テンプレートの item/user サンプルは採用しない
  - ES/CQRS：イベント記録は task と run のみ対象とする（意思決定は `task_decisions` で別管理）
  - イベントフロー MVP：DB スキーマと oRPC CRUD のみ（UI は後回し）
  - 司令官：apps/web 内の専用 Cursor Agent SDK で起票チャットを扱う
  - 司令室：apps/web が API と DB でタスクの状態と次の処理を決める
  - 作業実行：Worker はホスト上で Local Cursor SDK を使い、worktree を cwd として 1 run ずつ実行する
